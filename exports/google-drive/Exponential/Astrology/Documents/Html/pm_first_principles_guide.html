1. Embeddings, explained without engineering jargon

An embedding is a machine-readable fingerprint of text. It helps the system find related memories or reference material later.

Analogy

Embedding = a barcode for meaning

If the user says, "I am anxious about changing jobs," the system turns that sentence into a list of numbers. Later, when the user asks about career, the system can find that memory because the number-pattern is similar.

Why size matters

The number list has a fixed length

Some embedding models produce 1536 numbers. Some produce 1024. Some allow other sizes. The database search index has to know which size it is indexing.

The issue: The PRD says the provider is undecided, but the schema quietly assumed a 1536-number embedding shape. That is a hidden technical decision.
PM question Plain answer Decision
Does this mean we cannot use Claude? No. Claude can be the chat model, while embeddings come from a different provider. Separate "chat model" from "embedding model" in the architecture.
Why not just pick 1536 and move on? Because that would make the database depend on a vendor before the vendor decision is made. Keep embeddings in a separate table with model metadata.
Will this slow us down? A little more structure now prevents a painful migration later. Add the fast pgvector index after D5 is decided.
Memory text "I am changing jobs"
Embedding provider OpenAI, Voyage, Gemini, Cohere, or another choice
text_embeddings Stores vector plus provider, model, and dimension
What to tell engineering: D5 is open. Do not put `embedding vector(1536)` directly on `memories` or `jyotish_knowledge`. Use `model_catalog` plus `text_embeddings` instead.

2. Manual memory default, without chat friction

Manual memory means the system can notice useful facts, but the user chooses what becomes long-term memory. To avoid interrupting the live chat, ask memory questions at the end during CSAT/session review.

User says something "I am preparing for a promotion"
System queues a candidate It does not interrupt the conversation
End-of-chat review CSAT plus memory suggestions
Memory is stored Structured fact plus searchable embedding
Future answers improve Advice can reflect remembered context
PM-friendly position

Manual is a strong default

It is privacy-respecting and still useful. The best UX is to collect candidates silently and ask at the end, when the user is already reviewing the session.

Metric correction

Do not measure "opt-in" naively

If Manual is the default, "Manual or Auto opt-in rate" becomes confusing. Better metrics are candidate acceptance rate, memory saves per active user, and Manual retained after onboarding.

3. Model router, from first principles

A model router is like an air-traffic controller for AI calls. The product asks for a job, and the router decides which model should do it.

Answer

Yes, build the router

This is the right way to support OpenAI, Claude, Qwen, DeepSeek, Kimi, GLM, local models, hosted models, and different embedding providers without rewriting the product each time.

PM translation

One product, many model choices

The astrologer model, memory extractor, evaluator, and embedding model can all be different. We choose by quality, latency, cost, privacy, and licensing.

Job What the router can choose What we measure
Main astrologer Best reasoning/persona model Groundedness, tone, latency, cost
Memory extraction Cheaper structured-output model Precision, recall, user acceptance
Prediction evaluator Fast reviewer model Unsafe framing caught, false positives
Embeddings Separate embedding model Retrieval quality and dimension/index fit
Open-source Chinese model plan: valid direction, but run it through evals first. Treat Qwen, DeepSeek, Kimi, GLM, and successors as candidates behind the router, not as a schema-level commitment.

4. RAG, negativity, and prediction UX

RAG means the AI looks up relevant source material before answering. The point is not to remove hard material. The point is to make sure hard material is grounded, reviewed, and delivered responsibly.

Bad version

Raw text goes straight to the astrologer

  • Classical text is dumped into search.
  • The model retrieves alarming statements.
  • The guardrail has to catch the problem at the end.
Better version

Knowledge is reviewed before use

  • Sources are stored with provenance.
  • Chunks get topic, confidence, and delivery-risk labels.
  • Reviewed negative chunks can be used when relevant.
Raw source Book, report, expert note
Chunk Break into reviewable pieces
Review Label fatalistic, medical, financial, safe, etc.
Set delivery policy Normal, caution, transform, internal-only, or blocked
Answer Still passes output guardrails
Answer to "we will find a way to better manufacture predictions": yes, through deterministic chart engines, reviewed doctrine, repeatable evals, and versioned analysis outputs. The LLM can explain hard readings, but it should not freestyle ungrounded predictions.
Evaluator agent or system prompt? Use both. The prompt teaches the astrologer how to speak; the evaluator checks whether the answer actually did it. This is better UX because the user gets a natural answer, while the safety/review layer works in the background.

5. Costing: what PM owns vs what engineering must track

It is fine for business/product to own pricing and cost decisions. Engineering still needs the instrumentation so those decisions are based on facts.

Question Owner What must be recorded
How much should users pay? Business and product Plan tiers, subscriptions, one-off purchases, free limits.
Which model is worth the cost? Product plus engineering Provider, model, token use, latency, quality score, safety outcome.
Where is cost leaking? Engineering instrumentation Per-message cost, cache hit rate, retries, fallback usage, feature area.
PM takeaway: "We will take care of costing" is acceptable as a business decision. The product still needs cost telemetry from day 1 so you can actually take care of it.

6. How to support marriage matching and other future features

The key is not to build marriage matching too early. The key is to make the base able to support any future analysis.

Narrow approach

Add a marriage table now

This works for one future feature, but it bakes one use case into the foundation before the product has learned enough.

Recommended approach

Add a generic analysis layer

Each future analysis has a type, inputs, deterministic result, optional AI narrative, versions, and linked subjects.

Future feature What it becomes in the base model Why this stays flexible
Marriage matching analysis_type = compatibility, two subjects, deterministic match result. It can support Ashtakoot, Dashakoot, Manglik compatibility, or a future method.
Family charts Multiple birth_profiles linked to one account. The account can hold self, partner, child, parent, or other profiles.
Muhurta analysis_type = muhurta, one or more charts plus proposed date windows. Same table can store deterministic result and narrative explanation.
Persona upgrade New persona_version and regenerated profile marked active. Old outputs are superseded, not silently overwritten.
PM takeaway: marriage matching is a useful example, but the product base should support "any analysis involving one or more subjects." That keeps us pivot-friendly.

7. Why GitHub is better than Google Drive for this

Drive is good for sharing finished documents. Git is better for preserving the actual decision history.

History

Who changed what?

Git keeps the exact revision history. You can see what changed, when, and why.

Review

Discuss the diff

Instead of arguing inside a document, reviewers comment on the exact changed lines.

MCP

Machine-readable context

Markdown, SQL, JSON, prompts, and eval files are easy for tools and agents to index.

Write source Markdown, SQL, prompt files
Open PR Review the diff
Merge Decision becomes durable
Export HTML, PDF, Drive copy
Index MCP/tools read the canonical files
Operating rule: GitHub is the source of truth. Drive can hold exports, but not the only copy of a decision.

8. The schema changes in one PM-readable table

You do not need to understand every SQL line. You do need to know what each table protects.

Table or idea What it means Why it exists
model_catalog A registry of AI models and embedding models. Prevents hidden vendor assumptions.
text_embeddings Stores searchable numeric fingerprints separately from memories and knowledge. Lets us switch embedding providers later.
model_routes Stores which model should do which job. Lets us test open-source Chinese models without hard-wiring the product to one model.
memory_candidates Stores proposed memories until end-of-chat review. Keeps chat smooth while preserving user consent.
knowledge_sources and knowledge_chunks Stores raw Jyotish material and reviewable chunks with delivery policy. Allows serious readings without unreviewed raw text or harmful framing.
birth_profiles People or subjects connected to a user account. Supports self now and match/family subjects later.
analysis_runs A versioned record of one computed analysis. Supports compatibility, muhurta, profile regeneration, and pivots.
