System Design & Technical Architecture — Personal AI Astrologer
Status: Draft v0.1 Companion: 01_PRD_Personal_AI_Astrologer.md (the "what/why")
This document is the "how." It covers the whole system, the intelligence layers, every agent and sub-agent, the concrete system prompts, the data contracts, and the LLMOps/security/scaling practices. It's written so a novice can follow the moving parts and an engineer can build from it.

0. The single most important idea (read this first)
There are two kinds of intelligence in this product, and keeping them separate is what makes it work:
Deterministic intelligence (math). Computing the chart, the planetary positions, the dashas, the divisional charts, the doshas — this is solved astronomy + classical rules. It must be exact and never hallucinated. We compute it with code, store it as structured data, and treat it as ground truth.--tools


Generative intelligence (the LLM). Interpreting that chart in conversation, remembering the user, giving empathetic guidance. This is where the LLM lives.--general world models


The golden rule: The LLM reasons from the computed facts; it never invents them. When the astrologer says "your Jupiter is exalted in your 1st house," that fact came from the deterministic engine and was handed to the model — the model did not guess it. This is the difference between a trustworthy product and a confident-sounding random-fact generator.
A corollary: don't use an LLM where math or a lookup will do. Computing the current dasha is arithmetic, not a prompt. Fetching the D9 chart is a database read, not a prompt. We use LLMs only for synthesis, conversation, and extraction.

3. The Astrology Engine (foundation)
3.1 Build vs buy (Decision D1)
Option
Pros
Cons
In-house (Swiss Ephemeris)
Full control, exact, no per-call cost at scale, no third-party dependency/uptime risk, can compute everything (dashas, vargas, ashtakavarga…).
More engineering; licensing: Swiss Ephemeris is dual-licensed AGPL-3.0 or commercial — a closed-source commercial product needs the commercial license or full AGPL compliance. Budget/legal must clear this.
3rd-party API (various Vedic astrology APIs exist)
Fast to ship; offloads the math.
Per-call cost grows with scale; dependency on their uptime/accuracy/terms; less control; sending birth data to a third party (privacy).
Hybrid (recommended path)
Ship MVP on an API to validate product, migrate to in-house Swiss-Ephemeris engine before scale to control cost and data.
Two integrations over time.

Recommendation: Architect the engine behind a clean internal interface (AstrologyEngine.computeChart(birthData) → ChartObject) so the implementation (API vs in-house) is swappable without touching anything downstream. This is the key design move — everything else depends only on the Chart object, not on how it was produced.
3.2 What it computes (per PRD R-CHART-02)
Ascendant + lord · Moon sign/Sun sign · Nakshatra + Pada + lord · all grahas (Sun→Saturn, Rahu, Ketu; outers optional) with sign/degree/nakshatra/retro/combust/exalt/debilit/own + house placement · full Vimshottari (Maha→Antar→Pratyantar) + balance at birth · divisional charts (D1, D9 minimum; D2–D60 Shodashvarga) · doshas (Mangal, Kalsarpa, Sade Sati phase) · strengths (Ashtakavarga, Shadbala, Bhavabala) · yogas detected · ayanamsa (Lahiri) + house system recorded.
3.3 Temporal endpoint (R-CHART-05)
A separate method computeTemporalContext(chart, datetime=now) returns the currently active Maha/Antar/Pratyantar dasha, current Sade Sati phase, and key current transits. This is pure arithmetic over the natal chart and the date. Called at conversation time so guidance is time-relevant. The natal chart is immutable; this snapshot moves.
3.4 Validation
The uploaded reference chart (and any others) become golden fixtures. CI compares engine output (positions, dasha dates, dosha flags) to references within tolerance. This is both correctness QA and a guard against regressions when we swap implementations.

5. Chart rendering (visual kundali)
Input: the Chart object. Output: SVG (interactive-capable) + a PDF report.
Shared renderer: implement chart drawing as a pure function of the Chart object → SVG, packaged so web and mobile use the same logic (e.g., a shared TS rendering lib; React renders SVG natively, React Native via an SVG lib). This avoids three divergent chart implementations.
Styles: North-Indian (diamond) — default; South-Indian (square) — toggle. Divisional charts reuse the same renderer with different data slices.
Static vs interactive: charts are static once computed, so server-side SVG (cached in object store) is cheap; client-side SVG enables tap-to-inspect (R-VIZ-07). Recommend client-side SVG components fed by the Chart object, with a server-rendered PDF for download/sharing.
PDF report: generated async (job queue) and stored in object storage; user downloads a link. Mirrors the depth users expect from traditional reports.

7. The personalization pipeline (chart → system prompt)
This is the "permanent system instructions per user" you described. It has three stages.
7.1 Stage A — Chart Facts Sheet (L1, deterministic)
A template-generated, compact, authoritative rendering of the chart in a form the LLM can reason over without ambiguity. No LLM, no hallucination risk. Example fragment:
ASCENDANT: Cancer (Ashlesha pada 3), lord Moon.
MOON SIGN: Virgo. SUN SIGN: Capricorn. BIRTH NAKSHATRA: Chitra pada 2 (lord Mars).
PLANETS:
  Jupiter — Cancer, House 1, Exalted, Retrograde, lord of 6 & 9.
  Mars    — Scorpio, House 5, Own sign, lord of 5 & 10.
  ... (all grahas)
DOSHAS: Manglik — No. Kalsarpa — No.
KEY YOGAS: <detected>.

This is the non-negotiable ground truth injected into the astrologer's context.
7.2 Stage B — Interpretive Profile (L2, LLM, one-shot, stored)
An LLM (using the expert-astrologer persona — your uploaded astrologer prompt is an excellent base) reads the Chart object + Facts Sheet and produces a rich natal profile: personality, strengths/challenges, career/relationship/health themes, dasha narrative, dosha interpretation, remedies framing. Generated once, async, cached. This gives the astrologer depth and a consistent "reading" of the user without re-deriving it every message.
Optional sub-agents (depth): the synthesizer can be decomposed into specialists — Personality, Career, Relationships, Health, Dasha-narrative, Yogas/Doshas — each analyzing one slice, then a Composer merges them into the profile. This improves quality and parallelizes, at higher one-time cost. Start monolithic; split if quality demands.
7.3 Stage C — Runtime system prompt assembly (L3)
For each conversation, the Orchestrator assembles the astrologer's context as layers, ordered so the static part is cacheable:
[STATIC, cached per user]
  1. Persona + behavioral rules + SAFETY guardrails   (mostly identical across users)
  2. Chart Facts Sheet (L1)                            (static per user)
  3. Interpretive Profile summary (L2)                 (static per user)
[DYNAMIC, per request]
  4. Temporal context: current Maha/Antar/Pratyantar dasha, current transits, Sade Sati phase
  5. Retrieved memories (L4), clearly labeled "user-asserted, not chart fact"
  6. Recent conversation history (windowed/summarized)
  7. The user's new message

Critical optimization: sections 1–3 are static per user and identical across all their messages, so we exploit provider prompt caching — the large static prefix is cached and not re-billed/re-processed each turn. This is the single biggest cost/latency lever in the system (see §12).
Avoid context bloat: we do not stuff all 16 divisional charts, full dasha tables, and ashtakavarga into every prompt. The compact natal core covers ~90% of questions. For depth, the astrologer calls a tool (next section) to fetch exactly the slice it needs (e.g., D9 for marriage). This keeps prompts small, cheap, and focused.

8.2 The Astrologer (runtime agent) — the main brain
Role: answer the user's questions as a warm, knowledgeable Vedic astrologer, grounded in their chart + temporal context + memory.
Grounding: Chart Facts Sheet (authoritative) + Interpretive Profile (depth). Never invents chart facts.
Tools it can call (function calling):
getDivisionalChart(divisor) → e.g., D9 for marriage, D10 for career. (deterministic)
getDashaContext(datetime) → current/upcoming periods. (deterministic)
getYogasDoshas() / getStrength(planet|house) → ashtakavarga/shadbala slices. (deterministic)
searchMemory(query) → relevant user-asserted facts. (deterministic)
saveMemory(fact, category) → propose/store a memory (mode-gated).
(optional) searchJyotishKnowledge(query) → retrieve classical-rule references to ground interpretation (RAG, §13).
Output: streamed, empathetic, balanced, non-fatalistic, safety-compliant.
8.3 Profile Synthesizer (one-shot agent) — see §7.2.
8.4 Memory Extractor (async agent)
Role: read a conversation (or the latest turn) and extract salient, durable life facts (occupation, marital status, family, health context, finances, stated goals/worries). Categorize, assign confidence, dedupe/update against existing memory.
Model: a cheaper/faster model — extraction is simpler than the main conversation.
Mode gating: Off → don't run. Manual → produce candidates, surface for user confirmation, persist only on Save/Edit. Auto → persist directly (user can edit/delete later).
8.5 Deterministic tools (NOT agents)
Chart Query, Temporal Context, Memory Retrieve are plain functions/services. Treating them as "agents" would add latency, cost, and hallucination risk for zero benefit. Most "what does my chart say about X" requests resolve to: (a) figure out which data slice is relevant (the LLM decides via tool call) → (b) fetch it deterministically → (c) LLM interprets.
8.6 Guardrail / Safety layer — see §11. Wraps every LLM interaction; the persona prompt cannot override it.
8.7 End-to-end: one user message (the full turn flow)
1. User sends message  ──▶ API Gateway (auth, rate-limit, entitlement)
2. ──▶ Conversation Orchestrator
3.    Guardrail (input):   crisis/abuse/harm check.  If crisis → SAFE path, stop.
4.    Build context:
        • load cached static prefix (persona+facts+profile)   [prompt cache hit]
        • Temporal Context tool → current dasha/transits (now)
        • Memory Retrieve (if mode≠Off) → relevant facts
        • window/summarize recent history
5.    Call LLM (via Gateway) with tools enabled, streaming on.
6.    If model requests a tool (e.g., getDivisionalChart("D9")):
        • execute deterministically → return data → model continues.
7.    Stream tokens back to client as they arrive.
8.    Guardrail (output): scan final answer for medical/financial/legal/fatalism/safety.
        • If violation → rewrite/redirect before/while surfacing.
9.    Persist turn (transcript). Emit trace (prompt version, tools, tokens, latency, cost).
10.   Async: Memory Extractor runs (mode-gated). CSAT/thumbs hooks attached.

9.1 The Astrologer (runtime) — system prompt skeleton
ROLE
You are {AstrologerName}, a warm, wise Vedic astrologer (Jyotish). You give
personalized guidance grounded ONLY in the user's computed chart provided below.

ABSOLUTE RULES (override everything else)
- NEVER invent or alter chart facts. Every planetary position, dignity, house,
  dasha, yoga, or dosha you mention MUST come from the CHART FACTS or from a tool
  result. If you don't have a fact, fetch it with a tool or say you'll need to look.
- SAFETY: If the user expresses distress, hopelessness, or self-harm, STOP the
  astrology and respond with genuine care and crisis support. Never give methods.
- NO FATALISM: Never predict death, terminal illness, or catastrophe. Frame
  challenges as tendencies and growth, always preserving the user's agency.
- NO professional overreach: no medical diagnoses, no specific financial/trading
  directives, no legal advice. Gently redirect to qualified professionals.
- Treat MEMORIES as things the USER told you ("you mentioned…"), never as chart
  fact, and never let a memory override the chart.

STYLE
Empathetic, specific, balanced (≈60% strengths/opportunity, ≈30% challenges as
growth, ≈10% practical guidance/remedies framed as optional and low-cost).
Reference the actual chart ("As a Cancer ascendant with exalted Jupiter in the
1st…") so it feels personal, not generic.

TOOLS
- getDivisionalChart(divisor): use D9 for marriage/spouse, D10 for career, etc.
- getDashaContext(datetime): for any timing question.
- getStrength(target), getYogasDoshas(): for depth.
- searchMemory(query): to recall the user's life context.
- saveMemory(fact, category): propose remembering a salient life fact.

9.4 Guardrail — classification prompt skeleton (output check)
You are a safety reviewer. Given the astrologer's draft reply, flag if it:
- gives a medical diagnosis or treatment directive,
- gives specific financial/investment directives that risk loss,
- gives legal advice,
- predicts death/terminal illness/catastrophe or uses fatalistic doom framing,
- mishandles a user in crisis (should have provided support instead),
- contains other harmful content.
Return JSON: { "violation": bool, "categories": [...], "action":
"allow|rewrite|redirect", "note": "..." }.
DRAFT: {draft_reply}

(Pair the LLM check with fast deterministic classifiers/keyword + a dedicated crisis classifier on input, so the cheap checks run first.)

10. Memory system design (L4)
10.1 Storage model (hybrid)
Structured store (relational): known categories with current value (occupation, marital status, location) — easy to display, update, and reason over.
Vector store: free-form memories embedded for semantic retrieval ("user worried about a job change" surfaces when they ask about career).
Each memory record: id, userId, fact, category, embedding, source(conversationId/messageId), confidence, status(active|edited|deleted), createdAt, updatedAt.
10.2 Lifecycle
Extract (LLM, mode-gated) → Confirm (Manual only) → Store (structured + vector) → Retrieve (per turn) → Review/Edit/Delete (UI) → Erase (on account deletion or mode→Off).
10.3 Retrieval at conversation time
Combine semantic similarity to the user's message + recency + category relevance; inject the top-K, clearly labeled as user-asserted. Cap the count to control context size.
10.4 Modes (enforced server-side)
Off: extractor never runs; retrieval returns nothing; (optionally purge existing on opt-out).
Manual: extractor proposes; UI asks Save/Edit/Discard; persist only on user action.
Auto: extractor persists directly; user manages later.
10.5 Privacy (see §14)
Memories are sensitive PII — encrypted, user-controllable (view/edit/delete/export), included in erasure across both structured and vector stores and backups per policy.

11. Safety & guardrail layer (L5) — non-negotiable
Placement: in the LLM Gateway, wrapping every model call, so no persona prompt can disable it. Defense-in-depth: prompt-level rules and an independent output check.
Input checks (fast first): a dedicated crisis/self-harm classifier + abuse/harm checks before generation. Crisis → immediate supportive response with resources, persona suspended.
Output checks: scan drafts for medical/financial/legal overreach and fatalism/doom; rewrite or redirect before the user sees harm. (Stream-friendly: validate as it completes; hold/repair on violation.)
Categories enforced: self-harm/mental-health crisis, medical, financial, legal, fatalism/doom, exploitation/fear-selling, child safety, general harm (mapped to PRD §8 R-SAFE-*).
Logging: every safety event traced + flaggable by users (PRD R-FB-05) for review and eval.
Why it matters here specifically: classical sources contain alarming fatalistic text (death of relatives, disease, "suicide," etc., appear verbatim in standard reports). The product must systematically refuse to reproduce that and reframe constructively.

12.3 Prompt caching (the big lever)
The per-user static prefix (persona + Chart Facts + Interpretive Profile) is identical across all of that user's turns. Use provider prompt caching so it's processed/billed once, not every message. With a large grounding prompt and chatty users, this is a major cost and latency win. (Order the prompt so the static, cacheable part is the prefix and only the small dynamic tail changes.)
12.4 Reliability
Fallbacks: secondary provider/model on error or timeout.
Graceful degradation: memory/retrieval failure → still answer from the chart; tool failure → astrologer proceeds with the natal core and caveats.
Streaming + timeouts + retries with idempotency.
12.5 Data handling
Prefer providers/terms with no-training / zero-retention options for PII; minimize PII in prompts where feasible (PRD R-PRIV-06).

13. LLMOps & quality (L6)
Treat the AI like software: versioned, tested, observed, improved.
Prompt versioning: all system prompts (astrologer, synthesizer, extractor, guardrail) are versioned artifacts in source control. Per-user generated profiles record which generator version produced them, so we can regenerate on upgrades.
Tracing/observability: every turn emits a trace — assembled prompt (version), tools called + results, model + version, tokens in/out, latency (first-token + full), cost, guardrail outcome. (Tooling options: Langfuse, LangSmith, Arize Phoenix, Helicone, or equivalent.)
Evaluation harness:
Golden chart fixtures (the uploaded AstroSage chart + others) validate the deterministic engine (exact match within tolerance).
LLM-as-judge + curated rubrics score conversation quality (groundedness — did it stick to chart facts?; helpfulness; tone; safety).
Regression suite runs before any prompt/model change; block releases that regress groundedness or safety.
Red-team set for safety (crisis, medical/financial/legal bait, fatalism bait) — must pass.
Feedback loop: per-message thumbs + session CSAT + safety flags (PRD §6.7) feed dashboards, surface bad turns, and build eval sets / fine-tuning candidates over time.
A/B + flags: test prompt/model variants behind flags; ramp gradually; instant rollback.
Cost dashboards: per-feature token/cost, cache-hit rate, tier mix.
Optional Jyotish knowledge base (RAG): a curated corpus of classical rules/interpretations the astrologer can retrieve from, to deepen authenticity and reduce hallucinated interpretation. Grounds interpretation the way the Facts Sheet grounds facts. Build after MVP if quality needs it.

14. Security & privacy (engineering view)
Encryption: TLS in transit; encryption at rest for DBs, vector store, object store, backups. Consider field-level encryption for the most sensitive PII (birth data, memories).
Secrets/keys: managed secrets store; no provider keys client-side.
AuthN/Z: token-based sessions; per-user authorization on every data access (a user can only ever reach their own chart/memory/chats).
PII minimization: send the minimum necessary to providers; prefer no-retention terms.
Consent & control: explicit consent for memory; in-app export and full erasure that propagates to structured + vector + object stores + backups per retention policy.
Compliance: GDPR-style rights; India DPDP Act given the market; document data flows and a retention policy; data-residency awareness.
Abuse/rate limiting: at the Gateway, per user/IP, to protect cost and availability.

15. Scalability & reliability (engineering view)
Stateless services behind load balancers; horizontal autoscaling.
Async job queue for slow/heavy work: profile synthesis, memory extraction, PDF rendering, batch evals. Keeps user-facing latency low.
Caching: Chart object (immutable) cached aggressively; Interpretive Profile cached; prompt caching at the model layer; Redis for sessions + hot chart data.
Queues + workers decouple spikes (mass onboarding) from compute.
Backpressure & rate limits at the Gateway; provider-failover for LLMs.
Observability: metrics/logs/traces across services; alerting on latency, error rate, cost spikes, safety-flag spikes.
Cost at scale: the dominant cost is LLM tokens — controlled by tiering + prompt caching + small focused prompts (tools for depth, not giant contexts).

18. Voice agent (future) — architecture sketch
The brain stays the same; we add a voice I/O shell and optimize latency.
Mic ──▶ STT (streaming) ──▶ Conversation Orchestrator (SAME astrologer, tools,
                                memory, guardrails) ──▶ TTS (streaming) ──▶ Speaker

Reuse everything: same grounding, same memory, same safety — voice is a new interface, not a new brain.
Latency is the challenge: streaming STT, partial-response TTS, smaller/faster models or speculative responses, and tighter prompt caching matter even more.
Safety in voice: crisis detection on the transcript with the same override behavior; spoken responses must obey the same non-fatalism/medical/financial rules.
Designing the orchestrator as interface-agnostic now (text in/out as the contract) makes voice a clean add later.

19. Summary — the moving parts in one breath
We compute each user's chart deterministically (L0) and freeze it as a Chart object. From it we derive an authoritative Facts Sheet (L1) and an LLM-written Interpretive Profile (L2). At runtime, the Astrologer agent (L3) answers questions grounded in those facts, enriched by a live temporal/dasha snapshot and the user's memories (L4), protected by a guardrail layer (L5), with everything traced and improved via feedback + eval (L6). Deterministic tools fetch chart depth and memories on demand; a cheap Memory Extractor quietly builds the user model; the LLM Gateway makes models swappable and exploits prompt caching for cost. The same brain powers voice later. Two intelligences — math for truth, LLM for conversation — kept cleanly apart. That separation is the whole trick.
