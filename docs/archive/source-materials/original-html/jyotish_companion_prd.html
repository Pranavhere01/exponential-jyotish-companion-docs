01 Overview, problem & vision

What we're building, and why now

People who follow Vedic astrology today get generic, one-size-fits-all content — daily horoscopes by sun sign — or expensive, inconsistent human consultations they have to re-explain their life to every single time. We're building a personal AI astrologer: the user enters their birth details once, we compute their full kundali, and we generate an advisor that is permanently grounded in that specific chart and remembers the user across conversations.

The problem

A user wanting astrological guidance today either reads generic horoscopes (not personal), buys a static PDF report (not interactive, no follow-ups), or pays a human astrologer (costly, inconsistent, no memory between sessions). There is no product that is simultaneously grounded in the user's exact chart, conversational, consistent in quality, and able to remember the user's life over time. The desired state is an always-available conversational astrologer that knows your chart cold, remembers what you've told it, and gives consistent, personalized, non-fatalistic guidance for real decisions.

The bet

Giving each user an AI astrologer permanently grounded in their own computed chart, plus a memory of their life, will produce guidance personal and trustworthy enough to drive repeat engagement and willingness to pay. We'll know we're right when we see strong D7 / D30 retention, high session CSAT (≥ 4.2/5), and a healthy free → paid conversion among engaged users.

Why now

LLMs are finally good enough to reason over a complex structured chart without hallucinating the facts, hold a coherent multi-turn persona, and extract and use long-term memory. The astrology computation itself is a solved, deterministic problem (ephemeris math). The missing piece — turning a chart into a consistent, personalized advisor that remembers you — is now buildable.

05 Core user flows

How it feels to use

First run & onboarding

  1. User opens the app → sees the value prop and a "Create your chart" CTA.
  2. Authenticate (email, Google, or Apple).
  3. Birth-data form: full name, date of birth, time of birth (with a clear "don't know exact time?" path), place of birth (autocomplete → lat/long + timezone).
  4. User submits → loading state ("Casting your chart…").
  5. Backend computes the chart → generates the per-user astrological profile → renders the chart.
  6. User lands on the chart view — their visual kundali plus key highlights: ascendant, moon sign, nakshatra, current mahadasha, notable yogas/doshas.
  7. Clear CTA: "Talk to your astrologer."

Conversation

  1. User opens the chat with their astrologer.
  2. Optional first-time primer: the astrologer greets them by name and references one true thing from their chart.
  3. User asks a question — a life decision, a prediction, "what does my chart say about X".
  4. Astrologer responds — grounded in chart facts + current dasha/transit + any remembered context — with streaming text.
  5. User follows up; context persists within the session and, via memory, across sessions.
  6. Per-message feedback available; on session end, a CSAT prompt.

Memory capture mode-dependent

Off: nothing is stored beyond the chart and transcripts (retained per policy, not mined into a user model).  Manual: on detecting a salient fact, the system asks "Want me to remember this? Save / Edit / No" before persisting.  Auto: salient facts are extracted and saved automatically; the user can review, edit, or delete them anytime.

06 Functional requirements

The numbered spec

Every requirement is tagged with a priority and a stable ID, so engineering, design, and QA can reference it precisely and map it to a test case.

Authentication & account AUTH

R-AUTH-01MUSTSupport sign-up / sign-in via email + password, Google, and Apple (Apple is mandatory for the iOS App Store if other social logins are present).
R-AUTH-02MUSTA single account is consistent across web, Android, and iOS — same credentials yield the same profile, chart, memory, and history.
R-AUTH-03MUSTSecure session handling — token-based, refresh tokens, logout invalidates the session.
R-AUTH-04SHOULDAccount deletion that fully removes chart, memory, transcripts, and PII (right-to-erasure — see §8).
R-AUTH-05SHOULDEmail verification and password reset.
R-AUTH-06COULDPhone / OTP login (common and trusted in the India market).

Birth-data capture DATA

R-DATA-01MUSTCollect full name, date of birth, time of birth, and place of birth.
R-DATA-02MUSTPlace input uses autocomplete that resolves to latitude, longitude, and timezone — chart accuracy depends on this.
R-DATA-03MUSTTime of birth captured to the minute; the UI explains why exact time matters (ascendant and houses depend on it).
R-DATA-04MUSTA clear path for users who don't know their birth time (flag reduced confidence in time-dependent factors, or offer a "rectification later" note). The astrologer must be told when time is uncertain so it caveats time-sensitive claims.
R-DATA-05MUSTValidate inputs (plausible date ranges, valid time, resolved location) with inline errors.
R-DATA-06SHOULDAllow editing birth data — which triggers a full chart recompute and profile regeneration, with a warning that this resets chart-derived context.
R-DATA-07COULDSupport multiple saved profiles per account (e.g. family members) — single active profile in V1.

Chart generation & storage CHART

R-CHART-01MUSTCompute the chart server-side into a canonical structured Chart object. Computation is deterministic and cached / immutable per birth-data input.
R-CHART-02MUSTCompute, at minimum: ascendant & lord, moon/sun sign, nakshatra + pada; all grahas with dignity, retrograde/combust status and house placement; full Vimshottari timeline with balance at birth; D1 + D9 (broader Shodashvarga SHOULD); doshas (Mangal, Kalsarpa, Sade Sati); strengths (Ashtakavarga, Shadbala, Bhavabala); detected yogas.
R-CHART-03MUSTUse a defined ayanamsa (default Lahiri) and document the house system; expose ayanamsa in stored data for reproducibility.
R-CHART-04MUSTValidate engine output against known reference charts (the AstroSage fixture) to a defined tolerance.
R-CHART-05SHOULDRecompute time-sensitive elements (current dasha, transits, Sade Sati phase) relative to "now" at conversation time — not just at chart creation.

Chart visualization VIZ

R-VIZ-01MUSTRender a traditional North-Indian style Lagna chart and Navamsa chart from the Chart object.
R-VIZ-02MUSTDisplay a planetary-positions table and the Vimshottari dasha timeline with the current period highlighted.
R-VIZ-03SHOULDRender the full Shodashvarga set and strength tables, matching the depth of a traditional report.
R-VIZ-04SHOULDOffer South-Indian style as a toggle.
R-VIZ-05SHOULDGenerate a downloadable PDF report of the full chart + analysis.
R-VIZ-06MUSTCharts render correctly and legibly on web, Android, and iOS (shared rendering).
R-VIZ-07COULDInteractive charts — tap a house/planet for a detail popover.

The personalized astrologer CHAT

R-CHAT-01MUSTA chat interface where the AI astrologer answers questions grounded in their chart.
R-CHAT-02MUSTGrounded in the user's permanent astrological profile (generated once from the chart) so persona and chart facts persist across every session.
R-CHAT-03MUSTThe astrologer never invents chart facts. Positions, dignities, dashas, and yogas it cites must come from the computed chart, not the model's imagination.
R-CHAT-04MUSTResponses stream token-by-token.
R-CHAT-05MUSTIncorporates time context — knows the running dasha/antardasha and relevant current transits for timing questions.
R-CHAT-06MUSTIncorporates remembered context (per memory mode) so advice reflects the user's known life situation.
R-CHAT-07MUSTConversation history is persisted; users can revisit past chats.
R-CHAT-08MUSTTone is empathetic, balanced, and non-fatalistic — challenges framed as growth, never doom/death predictions.
R-CHAT-09SHOULDSupport multi-session threads vs one continuous thread. [Assumption] V1 = one primary continuous thread per user; multi-thread is a fast-follow.
R-CHAT-10SHOULDCan pull deeper chart data on demand (D9 for marriage, D10 for career) rather than holding the whole chart in context.
R-CHAT-11COULDSuggested prompt chips to reduce blank-page friction.

Memory MEM

R-MEM-01MUSTThree user-selectable modes: Off, Manual, Auto. [Assumption] default Manual — confirm in §13.
R-MEM-02MUSTMemories are user-asserted data points, not ground truth — referenced as "you mentioned…", never as astrological fact, and never overriding the chart.
R-MEM-03MUSTAuto: salient life facts (occupation, marital status, major events, stated goals) extracted and stored automatically.
R-MEM-04MUSTManual: surface a candidate memory and ask Save / Edit / Discard before persisting.
R-MEM-05MUSTOff: no user-model extraction or storage (beyond transcript retention per policy).
R-MEM-06MUSTA memory-management screen: list memories with category, source, and date; allow edit, delete, and export.
R-MEM-07MUSTMode changes respected immediately. Switching to Off offers to also delete existing memories.
R-MEM-08SHOULDMemories are categorized (occupation, relationships, family, health, finances, goals) for structured recall.
R-MEM-09SHOULDDe-duplication / update logic — a new "I got a new job" updates the occupation fact rather than piling duplicates.
R-MEM-10MUSTMemory data is encrypted and fully covered by account deletion / erasure.

Feedback & CSAT FB

R-FB-01MUSTPer-message thumbs up/down.
R-FB-02SHOULDA thumbs-down opens quick reason tags ("inaccurate", "too generic", "unhelpful", "felt wrong/unsafe") + optional free text.
R-FB-03MUSTPer-session CSAT (1–5) prompted on session end, with optional comment.
R-FB-04MUSTAll feedback stored and linked to the exact message/conversation + the prompt/model version used (for eval and improvement).
R-FB-05SHOULDA safety-flag path — user can report a response as harmful, routed to review with priority.

Settings & profile SET

R-SET-01MUSTView/edit birth data (with recompute warning), memory mode, account, and privacy controls.
R-SET-02MUSTData controls — export my data, delete my account/data.
R-SET-03SHOULDChart-style preference (North/South), language preference, notification preferences.

07 Non-functional requirements

Performance, scale, reliability

R-NFR-01MUSTChart computation completes within a few seconds at onboarding (cached thereafter).
R-NFR-02MUSTAstrologer first-token latency < 2s; full responses stream.
R-NFR-03MUSTApp remains responsive under load.
R-NFR-04MUSTDesigned for mass rollout — stateless services, horizontal scaling, async processing for heavy work (profile generation, memory extraction).
R-NFR-05MUSTCost controls — model tiering, prompt caching of the static per-user prompt, token budgeting.
R-NFR-06SHOULDTarget 99.9% availability for core chat + chart APIs.
R-NFR-07MUSTGraceful degradation — if the LLM provider is down, fail over to a secondary; if memory/retrieval fails, still answer from the chart.
R-NFR-08SHOULDWCAG-aligned accessibility (contrast, screen-reader labels, scalable text). Charts need text alternatives.
R-NFR-09SHOULDi18n-ready architecture (strings externalized). Hindi + English are the first targets for the India market.

08 Safety, ethics & compliance

First-class, not optional polish

A mass-market advisor that touches health, money, relationships, and life decisions can cause real harm if it overreaches or frightens vulnerable users. The classical texts our domain draws on contain fatalistic and alarming statements — predictions of death, disease — that appear verbatim in standard reports. We must not reproduce that.

Hard guardrails SAFE

R-SAFE-01MUSTMental health / self-harm: on distress, hopelessness, or self-harm ideation, the astrologer breaks persona and responds with care and appropriate crisis resources — no astrological platitudes, no fatalistic readings, no mention of methods. This overrides everything.
R-SAFE-02MUSTNo fatalism / no doom: never predict death, terminal illness, or catastrophe, and never frame a chart factor as an unavoidable curse. Challenges are tendencies and growth areas, with agency preserved.
R-SAFE-03MUSTMedical: no diagnoses, no medication/treatment directives. Health framed gently and redirected to qualified professionals; never weaponize chart "health" indications into alarming claims.
R-SAFE-04MUSTFinancial: no specific investment/trading directives that could cause loss. General, non-directive framing; recommend professionals for real decisions.
R-SAFE-05MUSTLegal: no legal advice; redirect to qualified counsel.
R-SAFE-06MUSTChild safety & abuse: standard prohibitions apply across the product.
R-SAFE-07MUSTNo exploitation: never use fear ("buy this remedy or disaster strikes") to manipulate; remedies are framed as traditional, optional, low-cost, never coercive.
R-SAFE-08MUSTDisclaimer: a clear, persistent disclaimer that the product is for guidance/entertainment and is not a substitute for professional medical, legal, financial, or psychological advice.
R-SAFE-09MUSTGuardrails apply on both input and output, enforced by a layer the persona prompt cannot override.

Data privacy & compliance PRIV

R-PRIV-01MUSTBirth data, memories, and transcripts are sensitive personal data — encrypted at rest and in transit.
R-PRIV-02MUSTExplicit consent for memory capture (especially Auto), with clear, granular controls.
R-PRIV-03MUSTRight to access, export, and erasure (GDPR-style; India's DPDP Act is directly relevant).
R-PRIV-04SHOULDData-residency awareness — plan for region-appropriate storage.
R-PRIV-05MUSTDefine a retention policy for transcripts and memories; honor deletion end-to-end, including vector stores and backups.
R-PRIV-06SHOULDBe deliberate about sending PII to third-party providers — prefer zero-retention / no-training terms; minimize PII in prompts.

09 Platforms & rollout

Where it runs, and how it ships

Platforms: web app, Android app, and iOS app, served by a single backend/API.

AssumptionCross-platform mobile (React Native or Flutter) to ship Android + iOS faster from one codebase, with a web app sharing as much logic and design as possible. Native is an option for maximum polish — decision in §13.

Phased rollout

  1. Internal alpha — team + trusted testers. Validate chart accuracy and conversation quality.
  2. Closed beta — invite cohort, feature-flagged. Tune prompts, guardrails, latency, cost.
  3. Staged GA — percentage rollout via flags, web-first or all-platforms by readiness, with monitoring and kill-switches.
  4. Scale — open to mass audience once safety, cost, and reliability targets hold.
Feature flags are a MUST for risky surfaces (memory Auto mode, new model versions, guardrail changes) so they can be toggled without redeploying.

10 Analytics & instrumentation

What we measure

Track — privacy-respecting and consented — the events needed to measure §2:

  • Onboarding funnel: form-start, each field, submit, chart-generated, first-message.
  • Engagement: sessions, messages/session, returning sessions, thread length.
  • Quality: per-message thumbs, session CSAT, safety flags, regeneration/edits.
  • Memory: mode chosen, candidates surfaced, saved/edited/discarded counts, memory-screen usage.
  • Reliability / cost: latency (first-token, full), token usage, model/version, provider, errors, fallbacks.
  • Every astrologer turn is traced with the assembled prompt version, tools used, and model version.

11 Dependencies

What we rely on

  • Astrology computation: an ephemeris library (e.g. Swiss Ephemeris) or a third-party astrology API — decision in §13.
  • LLM provider(s) for the astrologer, the memory extractor, the profile synthesizer, and embeddings.
  • Auth provider — build vs Firebase / Auth0 / Clerk / Cognito.
  • Geocoding / timezone provider for place → lat/long/tz.
  • Vector database for memory (and an optional Jyotish knowledge base).
  • Object storage for rendered charts/PDFs; a relational DB; a cache.

12 Monetization

Capability in scope, storefront out

The reference product's pattern is instructive: free basic chart → paid detailed reports → paid astrologer chat. Likely options for us:

  • Freemium: free chart + limited astrologer messages/day; subscription for unlimited chat, deeper reports, memory, premium features.
  • Subscription tiers (monthly / annual).
  • One-off purchases (full PDF report, specialized readings).
V1 MUST include the hooks — usage metering per user, entitlement checks — even if the storefront ships later. The pricing model itself is an open decision (§13).
