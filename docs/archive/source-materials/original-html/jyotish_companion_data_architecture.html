00 Orientation

The schema mirrors the system's two intelligences

Everything in this database falls on one side of a single line. On one side is deterministic ground truth — the computed Vedic chart, which must be exact and is never produced by a language model. On the other is generative output — the model's interpretations, conversations, and the longitudinal memory of the user. Keeping these apart in the data layer is what keeps the product trustworthy rather than a confident-sounding guess generator.

Gold · ground truth

Deterministic

Computed once by the astrology engine, stored immutably. The chart, planetary positions, dashas, divisional charts, doshas, strengths. The model reads these; it never writes them.

Amethyst · generated

Generative

The interpretive profile, the astrologer's messages, and extracted memories. Versioned by the prompt and model that produced them, so every claim is traceable back to its source.

The one structural difference from the reference schema. The platform this is modelled against is B2B and multi-tenant — every table carries a workspace_id. This product is B2C, so the tenant is the user. user_id is the universal scoping key, there are no workspaces, and a user can only ever reach their own rows.

01 Design principles

Six decisions that shape every table

User-scoped tenancy

user_id is denormalized onto every user-owned table. This makes Postgres Row-Level Security a one-line policy per table — the database itself enforces that a request can only ever touch its own user's data, exactly as the security model requires.

The chart is one immutable object

The computed chart is the central data contract: everything downstream — the facts sheet, the renderer, the runtime tools — reads it. It is stored once as charts.chart_object (JSONB), with only the natal-core scalars (ascendant, moon sign, nakshatra…) lifted out as columns for cheap display. Editing birth data does not mutate a chart; it computes a new one and supersedes the old, so any earlier grounded conversation stays coherent. We deliberately avoid pre-normalizing planets and dashas into separate tables — the deep slices are fetched by tools out of the JSON.

Memory is a hybrid in a single table

The memories table unifies two access patterns with pgvector: structured columns (category, status, supersedes_id) give the "current value" view, while the embedding column powers semantic recall. Memory modes — Off / Manual / Auto — are enforced server-side; a proposed row is a candidate awaiting the user's Save / Edit / Discard.

Right-to-erasure by construction

Every user-owned table cascades from users with ON DELETE CASCADE, so account deletion wipes all relational data in one operation. Vector-store, object-store, and backup erasure are handled out of band per the retention policy. Sensitive PII — birth data, memories, transcripts — is flagged for field-level (envelope) encryption at the app layer.

Enums for stable sets, text for churn

Native ENUM types model the stable domain sets (memory categories, message roles, safety categories, chart styles) for readability and integrity. Sets expected to evolve — feedback reason tags, plan codes — stay as text so they need no migration to change.

Heavy traces live outside Postgres

Per-turn observability detail belongs in an external tracer (Langfuse / LangSmith / Phoenix). The schema stores only the correlation keys on each message — trace_id, model_version, prompt_version — so feedback and the evaluation loop can tie any answer back to exactly what produced it, without a bloated traces table in the operational database.

02 The domains

Twenty-one tables, ten domains

The schema decomposes into clean functional groups. Each domain below maps to a phase of the product — from a user signing up, through casting and grounding their chart, to conversing, remembering, and staying safe.

Identity & access 4 tables

users · auth_identities · sessions · verification_tokens. Works with self-hosted or managed auth.

Birth data 1 table

birth_profiles. Name, date, exact time, resolved place — the only input the chart depends on.

The chart 3 tables

charts · interpretive_profiles · chart_artifacts. Immutable ground truth plus its derived readings and renders.

Conversation 2 tables

conversations · messages. One primary thread in V1; every turn records its versioning for feedback.

Memory 1 table

memories. Hybrid structured + vector store, mode-gated, with a de-dup supersede chain.

Feedback & safety 4 tables

message_feedback · session_csat · safety_events · safety_reports. Quality signal and the guardrail log.

Settings 1 table

user_settings. Memory mode, chart style, language, notification and privacy preferences.

Monetization 3 tables

plans · subscriptions · usage_counters. Metering and entitlement hooks; storefront ships later.

Knowledge (optional) 1 table

jyotish_knowledge. Curated classical-rules corpus for RAG; built post-MVP if quality needs it.

Analytics (optional) 1 table

analytics_events. Onboarding funnel and engagement; often offloaded to a product-analytics tool.

04 The Chart object

The one JSON contract everything reads

The renderer, the synthesizer, the runtime tools, and the LLM grounding all consume this single object. It is computed once, stored immutably in charts.chart_object, and cached. Keeping it as one contract — rather than scattering it across a dozen tables — is why the engine implementation (in-house vs API) stays swappable without touching anything downstream.

// shape stored in charts.chart_object (illustrative)
{
  "meta": { "ayanamsa":"Lahiri", "houseSystem":"WholeSign",
            "engineVersion":"1.4.0", "birthTimeKnown":true },
  "ascendant": { "sign":"Cancer", "nakshatra":"Ashlesha", "pada":3, "lord":"Moon" },
  "moonSign": "Virgo", "sunSign": "Capricorn",
  "planets": [
    { "graha":"Jupiter", "house":1, "dignity":"Exalted",
      "retrograde":true, "lordOfHouses":[6,9] }
    // … all grahas
  ],
  "divisional": { "D1":{}, "D9":{}, "D10":{} },   // up to D60
  "vimshottari": { "balanceAtBirth":{}, "mahadashas":[ /* Maha→Antar→Pratyantar */ ] },
  "doshas": { "mangal":{}, "kalsarpa":{}, "sadeSati":{} },
  "strengths": { "ashtakavarga":{}, "shadbala":{}, "bhavabala":{} },
  "yogas": [ { "name":"…", "planets":[], "houses":[] } ]
}
Why "current period" is not stored here. The active Maha / Antar / Pratyantar dasha, the Sade Sati phase, and live transits are pure arithmetic over the natal chart and now(). They are computed on demand at conversation time — the natal chart is frozen, the active periods move with the calendar. Short-lived caching lives in Redis, not in this table.

05 Layer mapping

How the schema maps to the intelligence layers

The System Design doc frames the product as six layers, L0–L6. Each lands somewhere concrete in this schema — or, deliberately, outside it.

LayerWhat it isWhere it lives
L0Computation → the Chart objectcharts.chart_object (immutable)
L1Grounding → the Chart Facts Sheetcharts.facts_sheet_text
L2Synthesis → the Interpretive Profileinterpretive_profiles
L3Conversation → the runtime Astrologerconversations · messages
L4Personalization → Memorymemories (structured + vector)
L5Safety → Guardrailssafety_events · safety_reports · messages.guardrail_action
L6Learning → feedback & evalmessage_feedback · session_csat + external tracer

The runtime astrologer (L3) is grounded by L1, enriched by L2, personalized by L4, protected by L5, and improved by L6 — and the schema records, on every message, the prompt and model version so that improvement loop has something to learn from.

06 Open decisions

Four choices that will tweak the schema

These are the product decisions from the PRD that touch the data layer directly. Worth settling before the first migration.

D1Astrology engine — build vs buy. Sets what engine_version means in practice, and whether birth data ever leaves your infrastructure (a privacy question, not just a cost one).
D3Default memory mode. The schema defaults user_settings.memory_mode to manual, following the PRD's assumption. Flip to Auto or Off if the privacy/personalization trade-off lands elsewhere.
D5LLM & embedding provider. Determines the vector dimension — the schema uses vector(1536). Change it to match your embedding model, or the pgvector indexes won't fit.
D8Chart depth at launch. D1 + D9 only vs the full Shodashvarga — changes how much computed data lives inside chart_object, and therefore the size of every cached chart.
